# Migration Authoring Guide

This guide covers how to write, test, and deploy database migrations for Tribunal.
All migrations are generated by Drizzle Kit from TypeScript schema definitions and
applied to Neon serverless PostgreSQL.

## Core Principles

### Append-only

Migrations are immutable once committed. Never edit an existing migration SQL file.
The migration journal (`drizzle/meta/_journal.json`) tracks each migration by its
SHA-256 hash. Editing a committed file breaks the hash chain and causes `db:check`
failures.

If a migration introduced a problem, fix it by generating a new forward migration.
See [Forward-Fix Strategy](#forward-fix-strategy) below.

### Idempotency

Every migration must be safe to run on a database where it has already been applied.
Drizzle Kit tracks applied migrations in `drizzle.__drizzle_migrations`, but manual
recovery scenarios and ephemeral branch testing depend on migrations not failing if
their effects already exist. Use `IF NOT EXISTS`, `IF EXISTS`, and conditional
`WHERE` clauses wherever possible.

### Backward Compatibility

Every migration must support the N-1 application version. When the migration runs
in production, both the old and new application code may be serving requests. This
means:

- New columns must have defaults (the old app will not supply values).
- Dropped columns must be unused by the current app before the drop migration ships.
- Renamed columns require a multi-phase approach (see patterns below).

### Performance Awareness

Every `ALTER TABLE` statement acquires an `ACCESS EXCLUSIVE` lock on the table. For
small tables this is instant. For tables with millions of rows, the lock blocks all
reads and writes until the statement completes. Choose migration patterns based on
table size and write traffic.

## Required Patterns

### Pattern 1: Column Addition

The safest schema change. Adding a column with a default does not rewrite the table
in PostgreSQL 11+ (the default is stored in the catalog).

```sql
-- Safe: column added with default, no table rewrite
ALTER TABLE "pull_request_state"
ADD COLUMN "review_count" integer DEFAULT 0 NOT NULL;
```

For nullable columns without a default, the operation is equally fast:

```sql
ALTER TABLE "pull_request_state"
ADD COLUMN "last_synced_by" text;
```

**Drizzle schema:**

```typescript
reviewCount: integer('review_count').default(0).notNull(),
lastSyncedBy: text('last_synced_by'),
```

### Pattern 2: Column Rename (Multi-Phase)

Column renames cannot be done atomically while maintaining backward compatibility.
Use a four-phase approach spread across separate deployments.

**Phase 1 -- Add new column and backfill:**

```sql
-- Migration: add-new-column-name
ALTER TABLE "repository" ADD COLUMN "display_name" text;

UPDATE "repository" SET "display_name" = "name" WHERE "display_name" IS NULL;
```

**Phase 2 -- Application dual-write:**

Deploy application code that writes to both columns. Reads switch to the new column.
No migration needed.

**Phase 3 -- Drop old column:**

After the dual-write deployment is stable and no rollback to Phase 1 code is needed:

```sql
-- Migration: drop-old-column-name
ALTER TABLE "repository" DROP COLUMN "name";
```

**Phase 4 -- Clean up application code:**

Remove dual-write logic. Only the new column remains.

### Pattern 3: Table Rename (Multi-Phase)

Table renames follow the same principle as column renames, but are rarer.

**Phase 1 -- Create new table:**

```sql
CREATE TABLE "template" (
  "id" integer PRIMARY KEY GENERATED ALWAYS AS IDENTITY,
  -- ... columns matching old table
);

INSERT INTO "template" SELECT * FROM "placeholder";
```

**Phase 2 -- Application dual-write:**

Application writes to both tables, reads from the new table.

**Phase 3 -- Drop old table:**

```sql
DROP TABLE "placeholder";
```

### Pattern 4: Backfill

Backfills populate existing rows with new data. They must be idempotent -- running
the backfill twice must produce the same result.

```sql
-- Idempotent: WHERE clause ensures already-backfilled rows are skipped
UPDATE "pull_request_state"
SET "review_count" = 0
WHERE "review_count" IS NULL;
```

For large tables, batch the backfill to avoid long-running transactions:

```sql
-- Batch backfill: process 10,000 rows at a time
-- Run this in a loop until 0 rows are updated
UPDATE "pull_request_state"
SET "review_count" = 0
WHERE "id" IN (
  SELECT "id" FROM "pull_request_state"
  WHERE "review_count" IS NULL
  LIMIT 10000
);
```

### Pattern 5: Index Creation

Always use `CONCURRENTLY` for indexes on tables with significant data. A regular
`CREATE INDEX` locks the table for writes during the entire build.

```sql
-- Safe: does not block writes
CREATE INDEX CONCURRENTLY IF NOT EXISTS "idx_pull_request_state_repository_id"
ON "pull_request_state" ("repository_id");
```

Note: `CREATE INDEX CONCURRENTLY` cannot run inside a transaction. Drizzle Kit
does not generate `CONCURRENTLY` by default. When Drizzle generates an index
creation migration for a large table, manually add `CONCURRENTLY` to the
generated SQL before committing. This is the one case where editing generated
SQL is acceptable -- but only before the migration is committed.

### Pattern 6: Column Drop

Dropping a column is destructive and irreversible. Verify the column is unused
before generating the drop migration.

**Pre-drop checklist:**

1. Search the codebase for all references to the column name.
2. Confirm no queries, ORM schema definitions, or API responses use the column.
3. Deploy the code change that removes all references.
4. Wait for the N-1 application version to be fully replaced.
5. Generate and apply the drop migration.

```sql
-- Only after confirming no application code references this column
ALTER TABLE "repository" DROP COLUMN IF EXISTS "legacy_status";
```

**CASCADE awareness:** If the column has dependent objects (indexes, foreign keys,
check constraints), use `CASCADE` to drop them automatically:

```sql
ALTER TABLE "repository" DROP COLUMN IF EXISTS "legacy_status" CASCADE;
```

Review what `CASCADE` will remove before using it. The dependent objects are gone
permanently.

### Pattern 7: Data Type Change (Multi-Phase)

Changing a column type can fail if existing data is incompatible, and `ALTER COLUMN
... TYPE` rewrites the entire table for most type changes.

**Phase 1 -- Add new column with target type:**

```sql
ALTER TABLE "event" ADD COLUMN "timestamp_tz" timestamp with time zone;

UPDATE "event"
SET "timestamp_tz" = "timestamp" AT TIME ZONE 'UTC'
WHERE "timestamp_tz" IS NULL;
```

**Phase 2 -- Application dual-write, read from new column.**

**Phase 3 -- Drop old column, optionally rename new column.**

### Pattern 8: Foreign Key Addition

Adding a foreign key constraint validates all existing rows, which scans the entire
table. For large tables, use `NOT VALID` to skip validation, then validate
separately.

```sql
-- Phase 1: Add constraint without validating existing rows (instant)
ALTER TABLE "pull_request_state"
ADD CONSTRAINT "fk_pull_request_state_repository"
FOREIGN KEY ("repository_id") REFERENCES "repository"("id")
ON DELETE CASCADE
NOT VALID;

-- Phase 2: Validate existing rows (can run concurrently with reads/writes)
ALTER TABLE "pull_request_state"
VALIDATE CONSTRAINT "fk_pull_request_state_repository";
```

The `VALIDATE CONSTRAINT` step acquires a `SHARE UPDATE EXCLUSIVE` lock, which
allows concurrent reads and writes but blocks other schema changes.

## Anti-Patterns (Never Do)

### Edit existing migrations

Once a migration is committed, it is immutable. Editing it breaks the journal hash
chain and causes CI failures. Generate a new forward migration instead.

### Add NOT NULL without a default

```sql
-- WRONG: Fails on tables with existing rows
ALTER TABLE "repository" ADD COLUMN "owner_id" integer NOT NULL;

-- CORRECT: Provide a default, or add as nullable and backfill
ALTER TABLE "repository" ADD COLUMN "owner_id" integer DEFAULT 0 NOT NULL;
```

### Drop columns before verifying they are unused

Always deploy the code change that removes column references before generating
the drop migration. The N-1 application version must not depend on the column.

### Rename without dual-write

```sql
-- WRONG: Breaks the N-1 application version immediately
ALTER TABLE "repository" RENAME COLUMN "name" TO "display_name";
```

Use the multi-phase rename pattern (Pattern 2) instead.

### CREATE INDEX without CONCURRENTLY on large tables

```sql
-- WRONG: Blocks all writes for the duration of the index build
CREATE INDEX "idx_event_created_at" ON "event" ("created_at");

-- CORRECT: Non-blocking index build
CREATE INDEX CONCURRENTLY "idx_event_created_at" ON "event" ("created_at");
```

### Non-idempotent updates

```sql
-- WRONG: Running twice doubles the value
UPDATE "counter" SET "value" = "value" + 1;

-- CORRECT: Idempotent with a guard clause
UPDATE "counter" SET "value" = 1 WHERE "value" = 0;
```

### Delete migrations from git history

The migration journal must be a complete, ordered history. Removing entries
causes `db:check` failures and breaks production deployments.

## Migration Checklist

### Before generation

- [ ] Schema follows project conventions (`timestamp with time zone`, `text` over
      `varchar(n)`, explicit `onDelete` on foreign keys)
- [ ] All foreign key columns have indexes
- [ ] New enums use `pgEnum` in `enums.ts`
- [ ] Column names follow `snake_case` convention (Drizzle maps to `camelCase`)

### After generation

- [ ] Review the generated SQL in `packages/database/drizzle/`
- [ ] Verify idempotency: would the migration succeed on a database where it
      already ran?
- [ ] Check for `ACCESS EXCLUSIVE` lock risk on large tables
- [ ] Add `CONCURRENTLY` to index creation on large tables if needed
- [ ] Backfills include `WHERE` guard clauses
- [ ] No `NOT NULL` additions without defaults

### Before merge

- [ ] CI migration job passes (the `migration` job in `.github/workflows/ci.yml`)
- [ ] Code review confirms safety of lock-taking operations
- [ ] Multi-phase migrations have follow-up tasks documented
- [ ] `bun run db:check` passes locally

## Testing Locally

Run the migration test suite to validate migrations against an ephemeral Neon branch:

```bash
bun run db:test-migrations
```

This command:

1. Creates an ephemeral Neon branch from the production database.
2. Applies the full migration history with `drizzle-kit migrate`.
3. Validates database invariants (table counts, migration counts, constraint checks).
4. Writes results to `packages/database/test-results/`.
5. Cleans up the ephemeral branch.

Required environment variables: `NEON_API_KEY`, `NEON_PROJECT_ID`, `DATABASE_URL`.

For a quick consistency check without a database connection:

```bash
bun run check:migrations
```

This verifies the migration journal integrity and schema-to-migration synchronization.

## Performance Considerations

### Large Table Operations (>1M rows)

Operations on large tables require extra care because they can hold locks for
extended periods or cause excessive I/O.

**Lock duration awareness:**

| Operation                       | Lock Type             | Impact                 |
| ------------------------------- | --------------------- | ---------------------- |
| `ADD COLUMN` (with default)     | `ACCESS EXCLUSIVE`    | Instant (catalog-only) |
| `ADD COLUMN` (no default, NULL) | `ACCESS EXCLUSIVE`    | Instant (catalog-only) |
| `DROP COLUMN`                   | `ACCESS EXCLUSIVE`    | Instant (catalog-only) |
| `ALTER COLUMN TYPE`             | `ACCESS EXCLUSIVE`    | Full table rewrite     |
| `CREATE INDEX`                  | `SHARE` lock          | Blocks writes          |
| `CREATE INDEX CONCURRENTLY`     | `SHARE UPDATE EXCL`   | Non-blocking           |
| `ADD CONSTRAINT (FK)`           | `SHARE ROW EXCLUSIVE` | Validates all rows     |
| `ADD CONSTRAINT NOT VALID`      | `SHARE ROW EXCLUSIVE` | Instant (no scan)      |
| `VALIDATE CONSTRAINT`           | `SHARE UPDATE EXCL`   | Scans, non-blocking    |

**Strategies for large tables:**

- Prefer `ADD COLUMN ... DEFAULT` over `ADD COLUMN` + separate backfill when the
  default is a constant. PostgreSQL 11+ stores the default in the catalog.
- Use `CREATE INDEX CONCURRENTLY` for all indexes on tables over 100K rows.
- Use `NOT VALID` + `VALIDATE CONSTRAINT` for foreign key additions.
- Batch backfills in chunks of 10,000-50,000 rows to limit lock duration and
  WAL generation.

### Neon-specific considerations

Neon's serverless architecture uses compute separation. Keep in mind:

- Long-running migrations keep the compute active and may increase costs.
- Ephemeral branches (used by `bun run db:test-migrations`) are cheap -- use them
  freely for local migration testing.

## Forward-Fix Strategy

Rollback is not supported. If a migration introduces a problem:

1. Identify the issue (wrong column type, missing index, bad default).
2. Edit the TypeScript schema to the correct state.
3. Generate a fix migration:
   ```bash
   bun run db:generate -- --name fix-describe-the-issue
   ```
4. Run preparation steps:
   ```bash
   bun run db:check && bun run check
   ```
5. Review the generated SQL and deploy.

## Workflow Summary

```
Edit TypeScript schema
        |
        v
bun run db:generate -- --name describe-change
        |
        v
Review generated SQL in packages/database/drizzle/
        |
        v
bun run db:check
        |
        v
Commit schema + migration together
        |
        v
Open pull request --> CI runs the migration job in ci.yml
        |
        v
Merge to main --> migration applies to production
```
